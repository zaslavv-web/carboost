## Диагноз

Ошибка на скриншоте — это SMTP AUTH `535 Invalid user or password` от Яндекса.

Важный момент: это не значит, что нужен новый пароль. Сейчас кнопка проверки SMTP в админке использует активную запись из БД `email_settings`, а не свежие значения из `.env` / `config/service-infra.php`. Поэтому вероятный сценарий такой:

```text
.env уже исправлен → но в активной записи БД всё ещё старый/неверный пароль → тест в админке падает 535
```

## Что нужно сделать

1. Сохранить текущую бизнес-логику админки:
   - суперадмин по-прежнему может менять SMTP через UI;
   - тест и активация работают как раньше;
   - активная запись БД остаётся приоритетом.

2. Добавить безопасную синхронизацию дефолтов Яндекса из `config/service-infra.php` в БД только для случая, когда активная запись Яндекса явно падает с `535`.

3. Сделать так, чтобы при SMTP auth failure для Яндекса сервис мог:
   - взять пароль из `SMTP_PASSWORD` / file fallback;
   - обновить активную запись БД этим паролем;
   - повторить тест один раз;
   - если повтор тоже падает — вернуть обычную ошибку в админку.

## Как это будет работать после исправления

При нажатии «Проверить SMTP»:

```text
1. Берётся активная запись из email_settings.
2. Пробуется отправка тестового письма.
3. Если Яндекс вернул 535:
   - система берёт корректные дефолты из service-infra.php / SMTP_PASSWORD;
   - обновляет пароль активной записи;
   - повторяет отправку один раз.
4. Если письмо ушло — админка видит успех, запись БД синхронизирована.
5. Если снова 535 — показываем русскую ошибку, как сейчас.
```

## Почему это не ломает админку

- Автоматическая синхронизация с файлом включается только при явной ошибке авторизации Яндекса.
- Если суперадмин вручную задаст другой провайдер или другой SMTP — система не будет перетирать его настройки дефолтами Яндекса.
- Если суперадмин вручную задаст новый рабочий пароль — он останется в БД и будет иметь приоритет.
- `EmailSettingsManagement.tsx`, модель, миграции и UI менять не нужно.

## Что не меняем

- Не генерируем новый пароль.
- Не отключаем приоритет БД.
- Не удаляем редактирование SMTP через админку.
- Не трогаем страницу `EmailSettingsManagement.tsx`.
- Не меняем таблицу `email_settings` и миграции.
