## Итог по ответам

- Заливаем в **уже существующую компанию** (владелец `growthpeak@yandex.ru`).
- Структура: **1 БЮ → 30 отделов → 119 линейных**. Раскладка сотрудников по отделам: 29 отделов × 4 + 1 отдел × 3 = 119.
- Механизм: **новая artisan-команда** в репозитории (существующий `demo:seed` не подходит — он создаёт *новую* компанию «ООО Демо» и тянет за собой магазин/риски/чаты, а нам нужно только 150 юзеров и оргструктура в реальной компании).
- Логины: реалистичные русские ФИО, рандом-пароли, CSV с логин/паролем на выходе.

## Что делаю

### 1. Новый файл `backend-laravel/app/Console/Commands/SeedOrg150.php`

Сигнатура:

```
php artisan org:seed-150
    --owner-email=growthpeak@yandex.ru
    --bu-name="Основной бизнес-юнит"
    --departments=30
    --headcount=150
    --email-suffix=@demo.growth-peak.pro
    --marker=seed150
    --password-length=12
    --reset      # снести ранее засеянные строки этой же меткой
    --dry-run    # ничего не пишет, только показывает план
```

**Что делает:**

1. Находит `company_id` по email владельца → `users.email` → `profiles.company_id`. Прерывается, если не нашла — не хочу засеять чужую компанию.
2. При `--reset` — удаляет только предыдущие seed-строки: пользователи, у которых `email LIKE '%+seed150@demo.growth-peak.pro'`, и связанные `profiles`, `user_roles`, `team_members`, `departments.head_user_id=NULL`. Реальные пользователи не трогаются.
3. Создаёт 1 родительский `departments` (БЮ) + 30 дочерних `departments` c `parent_id = BU_id`.
4. Создаёт `positions`: «Руководитель БЮ», «Руководитель <название>» ×30, 6 линейных позиций (по 1 на несколько отделов, чтобы должности не были одинаковыми: «Специалист», «Ведущий специалист», «Аналитик», …).
5. Через `AuthUserService` создаёт 150 юзеров:
  - **1** с ролью `hrd` (руководитель бизнес-юнита) → head of БЮ, позиция «Руководитель БЮ».
  - **30** с ролью `manager` → по одному в каждый отдел + `departments.head_user_id`.
  - **119** с ролью `employee` → 4 на отдел (последний отдел получит 3), + `team_members(manager_user_id, member_user_id)`.
6. ФИО: два пула — 60 мужских + 60 женских имён, 120 фамилий (из русского корпуса). Логин: `translit(firstname).translit(lastname)+seed150@demo.growth-peak.pro` c числовым суффиксом при коллизии.
7. Пароль: `Str::password(12, letters:true, numbers:true, symbols:false)` — на прод в открытом виде не хранится, только в CSV на выходе.
8. Каждый юзер: `profiles.is_verified=true`, `hire_date` рандомная в пределах последних 5 лет, `avatar_url` — Dicebear seed по логину, `department`, `position`, `position_id` проставлены.
9. Всё внутри одной `DB::transaction()`. При исключении — откат.
10. На выходе — CSV `storage/app/demo-seed-150-<timestamp>.csv` c колонками: `email,password,full_name,role,department,position,is_head`.

### 3. Крайние случаи, которые обрабатываю

- Уже есть люди в компании (реальный `growthpeak@yandex.ru` + возможные другие) — они не трогаются.
- Если у company_admin (`growthpeak@yandex.ru`) в компании уже сколько-то отделов — новые 30 добавятся рядом. При `--reset` удаляются только seed-отделы (по name-префиксу `[seed150]` в скрытом поле или по `parent_id = наш БЮ`).
- MySQL/PG различия обходятся Eloquent-запросами (`DB::table` без DB-специфичного SQL).
- Идёт валидация `AuthUserService`: если email уже занят реальным юзером — суффикс сдвигается.
