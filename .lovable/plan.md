## Цель

Запустить новый модуль **«Трекер»** внутри Career Track: единое место руководителя для OKR, поручений и встреч 1:1 со связями «поручение ↔ цель» и базовой аналитикой здоровья. Без интеграции с Jira и без HR-триггеров на этой итерации (заложим заделы в схеме, но автоматику не подключаем).

## Что входит в MVP (полный каркас сущностей)

Три сущности по схемам BPMN из материалов + матрица привязок:

1. **Цели (OKR)** — жизненный цикл из схемы 1: `draft → published → needs_review → archived`. Поля: название, описание, держатель, период (квартал), ключевые результаты (KR) с весом и текущим прогрессом 0–100%, опциональная команда/проект.
2. **Поручения (Tasks)** — жизненный цикл из схемы 2: `draft → published → awaiting_checkin → done | orphan | needs_attention | archived`. Атрибут **срочности** (critical / high / medium / low) — обязателен. Привязка к 0..N целям (вес влияния на KR). Поле под Jira-ключ (пока ручное, без синхронизации).
3. **Встречи 1:1** — расписание между руководителем и сотрудником, повестка (список пунктов, можно тянуть открытые поручения), заметки, договорённости (создают поручения в один клик), статус: `planned / done / cancelled`.

## Точка входа и навигация

Новый верхнеуровневый пункт сайдбара **«Трекер»** (иконка `Target`) с под-разделами:

```text
Трекер
├─ Мои цели          /tracker/goals
├─ Поручения         /tracker/tasks
├─ Встречи 1:1       /tracker/one-on-ones
└─ Дашборд           /tracker/dashboard  (Manager/HRD)
```

Раздел виден всем 5 ролям, но содержимое зависит от роли (см. ниже).

## Доступ по ролям

| Роль | Что видит |
|---|---|
| **Employee** | Свои OKR (как держатель), назначенные поручения, свои 1:1. Может создавать поручения себе и обновлять прогресс KR. |
| **Manager** | Всё, что Employee + OKR/поручения подчинённых, проводит 1:1, создаёт поручения подчинённым, видит дашборд своей команды. |
| **HRD** | Глобальный дашборд по компании: % каскада, % OKR с чек-инами, доля привязанных поручений, индекс здоровья. Read-only по сущностям. |
| **Company Admin** | Настройки модуля для компании (периоды OKR, шаблоны, правила срочности). Полный read-доступ. |
| **Superadmin** | То же + кросс-компанийный обзор. |

Multi-tenancy через `company_id` на всех таблицах + RLS, как в существующих модулях.

## Ключевая фича B: «Связка поручение ↔ цель в один клик»

- В карточке поручения — селект «Привязать к цели» с поиском по активным OKR держателя/команды.
- В карточке OKR — список привязанных поручений с прогрессом и срочностью.
- На дашборде — метрика **«% поручений, привязанных к OKR»** с динамикой по неделям.

## Дашборд

- **Manager:** воронка каскада своей команды, список «требующих внимания» поручений (по матрице срочности), просроченные KR, ближайшие 1:1.
- **HRD:** агрегаты по компании, тепловая карта по руководителям, индекс здоровья (композитная метрика: каскад × чек-ины × привязка × своевременность).

Графики на Recharts (как в существующей HRD-аналитике).

## Технические детали

### Новые таблицы (Supabase, RLS + GRANT по проектному стандарту)

- `tracker_okr_periods` — справочник периодов (квартал/год) на компанию.
- `tracker_goals` — цели OKR. `status enum`, `holder_id`, `period_id`, `team_id?`, `progress` (computed из KR).
- `tracker_key_results` — KR одной цели, `weight`, `current_value`, `target_value`, `unit`.
- `tracker_tasks` — поручения. `status enum`, `urgency enum`, `assignee_id`, `author_id`, `due_at`, `jira_key?`, `parent_task_id?`.
- `tracker_task_goal_links` — m:n привязка task↔goal с `impact_weight`.
- `tracker_task_checkins` — история чек-инов и обновлений статуса.
- `tracker_one_on_ones` — встречи, `manager_id`, `employee_id`, `scheduled_at`, `status`.
- `tracker_one_on_one_agenda` — пункты повестки + связь с поручением/целью.
- `tracker_audit_log` — журнал смен статусов (требование из схемы 1: «ни одна сущность не удаляется молча»).

Все таблицы — с `company_id` и RLS-политиками через существующий `has_role(_user_id, _role)` + проверку «manager видит подчинённых» (используем `team_members` / `profiles.manager_id` — уточню при имплементации).

### Что НЕ делаем в этой итерации (отложено)

- Реальная синхронизация с Jira (поле `jira_key` пустое, кнопка-заглушка).
- HR-триггеры на увольнение/перевод (статус `needs_review` доступен вручную).
- Автоматические эскалации по матрице срочности через cron (UI показывает таймер, но не шлёт уведомления). Заложим структуру под Edge Function — включим следующей итерацией.
- Шаблоны OKR (модуль E из документа — низкий приоритет).

## Порядок реализации

1. **Миграция БД** — все 8 таблиц с RLS/GRANT + enum-типы.
2. **Навигация** — добавить пункт «Трекер» в сайдбар, 4 роутa.
3. **Цели (OKR)** — CRUD, список, карточка с KR и прогрессом.
4. **Поручения** — CRUD, матрица срочности, статусы, фильтры.
5. **Связка task↔goal** — диалог привязки, метрика на дашборде.
6. **1:1** — расписание, повестка, конвертация пункта в поручение.
7. **Дашборд** — Manager-view, затем HRD-view на Recharts.
8. **Аудит-лог** — триггеры на смену статуса.
