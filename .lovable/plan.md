## Что на самом деле ломается

Сценарий: пользователь вошёл в кабинет → закрыл вкладку → открыл сайт заново → браузер поднимает приложение с сохранённым `localStorage`/`sessionStorage` → экран становится чёрным. Очистка истории за час помогает, потому что удаляет сохранённое состояние браузера, после чего сайт стартует как новый гость.

Это не похоже на кеш чанков. Главная подозрительная зона в коде — **загрузка приложения с сохранённым `laravel_token` до того, как React успел безопасно восстановить пользователя**.

## План исправления

1. **Убрать ложное восстановление через `main.tsx`**
   - Оставить recovery только для реальных ошибок загрузки JS-чанков.
   - Не чистить auth-сессию из `main.tsx`: сейчас это может маскировать проблему и создавать непредсказуемое поведение.

2. **Сделать отдельный режим cold-start восстановления сессии**
   - В `LaravelAuthContext` явно хранить статус: `checking`, `authenticated`, `guest`, `failed`.
   - При старте с сохранённым `laravel_token` показывать нормальный экран “Восстанавливаем сессию…”, пока `/auth/me` не вернёт результат.
   - Если токен невалиден/протух/сервер вернул HTML/timeout/network error — очистить только auth-ключи и перевести пользователя на `/login`, а не оставлять чёрный экран.

3. **Починить главную причину чёрного экрана: провайдеры монтируются слишком рано**
   - Сейчас `ImpersonationProvider`, `BrandingProvider`, `ChatProvider`, `AnalyticsBootstrap` монтируются сразу внутри `AuthProvider` даже на этапе восстановления сохранённой сессии.
   - Ввести внутренний `AuthenticatedAppProviders`: монтировать эти провайдеры только когда auth boot завершён.
   - Для публичных страниц (`/`, `/pricing`, `/login`, `/reset-password`) не запускать кабинетные провайдеры вообще.

4. **Очистить битое persisted-состояние при старте**
   - Если вкладка была закрыта во время impersonation или остались старые `sessionStorage` ключи, они не должны применяться к новой вкладке.
   - При cold-start валидировать: есть `laravel_token` → только после успешного `/auth/me` разрешать использовать user/profile/roles.
   - Если токена нет — удалить остаточные impersonation-ключи.

5. **Заменить зависание на диагностируемый экран**
   - В `ProtectedRoute` добавить состояние “Сессия не восстановилась” с кнопкой “Очистить сессию и войти заново”.
   - Это убирает ситуацию, где пользователь видит просто чёрный экран и единственный способ — чистка истории браузера.

6. **Проверка именно вашего сценария**
   - Зайти в кабинет.
   - Закрыть вкладку.
   - Открыть сайт заново.
   - Проверить два варианта:
     - сохранённый токен валиден → кабинет открывается;
     - сохранённый токен невалиден/протух → пользователь попадает на `/login` без чёрного экрана и без ручной очистки истории.
