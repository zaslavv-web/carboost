## Ответ на вопрос

**Ручной запуск не нужен в норме** — автодеплой по `push` в `main` остаётся основным механизмом. `workflow_dispatch` я добавил только как кнопку "Re-run" для диагностики: открыть GitHub → Actions → "Deploy to server" → Run workflow. Это не замена авто-деплоя, а инструмент отладки на случай, когда нужно пересобрать без нового коммита.

Ты прав: цель — минимум ручного труда. Поэтому план ниже — про то, чтобы автодеплой действительно доезжал до сервера сам, а ручная кнопка просто **была доступна** на крайний случай.

## План

### 1. Диагностика — почему авто-деплой "успешен" в GitHub, но файлы не на сервере

Нужно посмотреть логи последнего прогона Actions, конкретно:

- Зелёный или красный был `build-frontend` / `build-backend`?
- Что вывел шаг `Write frontend deploy marker` и шаг rsync (сколько файлов передано, в какой `TARGET_DIR`)?
- Что отдаёт `curl https://growth-peak.pro/deploy-version.txt` — старый commit или новый?

Без этих 3 ответов любой фикс — гадание. Это **не ручной труд по деплою**, это один раз глянуть лог.

### 2. Типовые системные причины (проверю по логам)

- rsync ходит не в тот каталог (TARGET_DIR указывает на staging, а nginx отдаёт другую папку).
- nginx/OPcache кэширует старые файлы → нужен `opcache_reset` после деплоя.
- На сервере симлинк `current → releases/xxx` не переключается.
- SSH-ключ/known_hosts отвалился после смены host key на nic.ru → rsync молча падает, но job помечен зелёным (нет `set -e`).

### 3. Фикс (после диагностики)

- Починить конкретную причину из п.2.
- Добавить в конец workflow **автоматическую проверку**: `curl https://growth-peak.pro/deploy-version.txt` и сравнить commit с `GITHUB_SHA`. Если не совпало — job красный. Это закроет ситуацию "Actions зелёный, а сайт старый" навсегда, без ручных проверок.
