### 2. Классифицировать причину

По коду ответа:

```text
401  →  Sanctum не принимает токен: истёк, стёрт, не тот guard, или PAT удалён
419  →  Session/CSRF: web-guard вместо api, cookie SameSite/domain, время
500 + текст "Unauthenticated" в JSON  →  bug в middleware/policy, выкидывает 401 по ошибке
таймаут (AbortError) → PHP-FPM висит на конкретном endpoint
```

### 3. Починить причину, не симптом

Возможные исправления по итогу шага 2 (реализую только тот вариант, который подтвердится):

- **Ложные 401 из policy/authorize** — поправить конкретный контроллер/policy, чтобы возвращать 403 (доступ), а не 401 (нет сессии). 401 должен означать только «токена нет/не валиден».
- **Sanctum PAT протухает** — увеличить `sanctum.expiration` или обновлять `last_used_at` (сейчас может истекать по неактивности).
- **419 на api-роуте** — убрать `VerifyCsrfToken`/`StartSession` из api-группы; проверить `bootstrap/app.php` и `config/sanctum.php` `stateful` домены.
- **Таймауты на конкретном RPC** — оптимизировать запрос или увеличить лимит в `client.ts` только для тяжёлых операций; не сбрасывать сессию по `AbortError` (текущая логика слишком агрессивна).
- **Клиентская защита от ложных выходов** — при 401 сначала попробовать `POST /auth/refresh` (если бэк умеет) или `GET /auth/me`; только при повторном 401 звать `clearSession`. Это одна маленькая правка в `client.ts` + `LaravelAuthContext`, снимающая класс проблем «одна ошибка → всех выкинуло».

### 4. Верификация
