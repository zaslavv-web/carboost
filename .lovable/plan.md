
## Цель

Развернуть тестовую компанию **ООО «Демо»** (~150 пользователей, единый пароль `DemoPass!2026`, максимально наполненный контент по всем модулям), чтобы (а) выявить и починить баги на реалистичном объёме данных и (б) иметь готовое окружение для демонстраций клиентам. Оставляем оба способа запуска — UI и artisan.

## Как запустить (что уже есть)

**Вариант A — интерфейс (рекомендуется для демо):**
1. Зайти под superadmin.
2. Открыть `/superadmin/demo-seed`.
3. Нажать **«Создать / дополнить»** (по желанию — задать headcount).
4. Дождаться статуса, посмотреть счётчики (users / tasks / orders / …), скопировать список логинов кнопкой «Скопировать все».
5. Разлогиниться и войти любым логином из списка с паролем `DemoPass!2026`.

**Вариант B — консоль на сервере (для CI / быстрого ресета):**
```bash
php artisan demo:seed                 # создать/дополнить
php artisan demo:seed --headcount=150 # указать численность
php artisan demo:seed --reset         # снести и пересоздать
```
Идемпотентно, дубли не создаются.

## Что должно быть видно в интерфейсе после сидинга

- **Superadmin:** «ООО Демо» в списке компаний, ~150 пользователей, возможность impersonate.
- **HRD / Company Admin:** оргструктура (8 департаментов, 20+ должностей), карьерные треки и назначения, дашборд аналитики с ненулевыми `risk_scores` / `comfort_scores`, инициативы с голосами.
- **Руководитель:** подчинённые, задачи, оценки 360°, peer-recognition.
- **Сотрудник:** цифровой паспорт, задачи, магазин с товарами и заказами, баланс валюты и транзакции, чаты, уведомления, достижения.

## План работ

Так как реального запуска ещё не было, действуем итеративно: сначала прогоняем как есть, ловим ошибки, чиним категориями.

**Шаг 1. Прогон сидинга и сбор ошибок**
- Запустить сидинг из UI под superadmin.
- Собрать: HTTP-ответ эндпоинта, логи Laravel, счётчики после прогона.
- Параллельно проверить `php artisan demo:seed` на сервере (тот же код, другой контур).

**Шаг 2. Разбор ошибок по категориям**
Типичные точки поломки на 150 юзерах, которые чиним пачкой, а не по одной:
- нарушения уникальных индексов (email, `user_id + role`, slug должности);
- отсутствующие FK / `company_id` в связанных таблицах;
- триггеры/наблюдатели, падающие на bulk-insert (уведомления, аудит);
- памяти/таймаута — переводим тяжёлые куски на `chunk()` / очереди;
- расхождения схемы (колонки, которых ждёт сидер, но их нет в миграциях).

**Шаг 3. Пройти по ролям в UI**
Логинимся тремя-четырьмя типовыми пользователями (сотрудник, руководитель, HRD, superadmin) и проверяем ключевые экраны:
- дашборды ролей, оргструктура, треки, задачи;
- магазин + оформление заказа, баланс и транзакции;
- аналитика HRD (Recharts не должен падать на пустых сериях);
- чаты, уведомления, достижения;
- страницы, где раньше были жалобы (мобильные таблицы, presentation, ассессмент).

Всё, что визуально ломается на реальном объёме (пустые состояния, переполнения, N+1-долгие ответы) — чиним в этом же проходе.

**Шаг 4. Проверка ресета**
Прогнать `--reset` из UI и из консоли — убедиться, что данные ООО «Демо» полностью удаляются и создаются заново без ошибок FK.

**Шаг 5. Финал**
- Обновить README-блок по демо-контуру (как запускать, дефолтный пароль, где страница управления).
- Оставить страницу `/superadmin/demo-seed` доступной только superadmin.
