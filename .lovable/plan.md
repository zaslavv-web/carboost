Проблема похожа не на кеш ассетов, а на сломанную/устаревшую сохранённую авторизацию: при повторном заходе фронт видит старый `laravel_token`, стартует личный кабинет, параллельно поднимает профиль/роли/брендинг/чаты/уведомления, и где-то в этой цепочке приложение зависает или падает до нормального редиректа.

План исправления:

1. Усилить клиентский auth-boot
   - В `LaravelAuthContext` сделать явное состояние инициализации: сначала прочитать токен, проверить `/auth/me`, только потом разрешать защищённым страницам грузить личный кабинет.
   - Любой невалидный, битый, устаревший или нераспознаваемый токен удалять централизованно.
   - Не оставлять приложение в `loading=true`, даже если `/auth/me` вернул HTML, 401, 419, 500 или network error.

2. Добавить безопасную очистку auth-состояния
   - Ввести единую функцию очистки: `laravel_token`, временные OAuth/social-signup ключи, impersonation-ключи, query-cache пользовательских данных.
   - Не чистить всё подряд, а только auth-related storage, чтобы не ломать тему/настройки интерфейса.
   - После очистки отправлять пользователя на `/login` вместо чёрного экрана.

3. Закрыть cookie/Sanctum-несостыковку
   - Сейчас комментарии в коде говорят «cookies не нужны», но симптом именно повторной сессии требует учесть cookie/credential edge cases.
   - В клиенте добавить `credentials: "same-origin"` для локального `/api` и безопасное поведение для внешнего API, чтобы браузер не зависал между cookie- и bearer-режимами.
   - На ошибках 401/419 явно считать сессию протухшей.

4. Защитить провайдеры личного кабинета
   - `BrandingProvider`, `ChatProvider`, `useUnreadNotifications`, `useUserProfile/useUserRoles` должны стартовать запросы только после завершённой auth-проверки и наличия валидного пользователя.
   - Ошибки вторичных запросов не должны валить весь личный кабинет.

5. Сделать recovery видимым, а не «чёрным»
   - Если сохранённая сессия повреждена, показать короткий экран восстановления/редиректа на логин.
   - `ErrorBoundary` дополнить кнопкой «Выйти и очистить сессию», чтобы пользователь сам мог восстановиться без сброса кеша браузера.

6. Проверка
   - Проверить сценарии: первый логин, refresh `/dashboard`, закрыть/открыть вкладку, повторный вход со старым токеном, протухший/битый токен.
   - Убедиться, что неавторизованный пользователь уходит на `/login`, а авторизованный попадает в кабинет без зависания.
