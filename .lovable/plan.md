## План

1. **Проверить, какой код реально работает на сервере**
   - Использовать существующий публичный эндпоинт `https://growth-peak.pro/api/diag`.
   - Сравнить поле `commit` с текущей версией репозитория.
   - Если `commit` старый или `unknown`, значит авто-синхронизация/деплой не применили последние изменения.

2. **Сделать безопасный trigger-change для backend deploy**
   - Добавить новый небольшой маркер версии в уже существующий `/api/diag`, чтобы после деплоя было видно, что именно этот коммит доехал до сервера.
   - Не полагаться на новый `_route_check`, потому что он уже показал 404 и не помогает, если код не обновился.

3. **Усилить деплой-скрипт**
   - Оставить `route:clear` перед `route:cache`.
   - Убрать дублирующий `event:cache`.
   - При необходимости добавить вывод `php artisan route:list | grep admin/users` в deploy log, чтобы на сервере было видно, зарегистрирован ли маршрут после кеширования.

4. **После подтверждения деплоя**
   - Проверить `https://growth-peak.pro/api/diag`: там должен появиться новый маркер.
   - Затем проверить назначение компании в интерфейсе.
   - Если `diag` обновился, но PATCH всё ещё 404 — проблема уже не в git sync, а в серверном routing/nginx/другой директории Laravel.

5. **Очистка временной диагностики**
   - Удалить `/admin/_route_check` после диагностики.
   - Удалить временный redeploy/debug marker.
   - Вернуть обработку ошибки на фронте к обычному сообщению, когда backend route подтверждён.
