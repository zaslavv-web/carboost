# ADR-001 — Frontend живёт в корне репозитория, `core` — в `backend-laravel/`

**Дата:** 2026-07-13
**Статус:** Принято
**Контекст:** реструктуризация в монорепо `apps/web` + `core` + `services/*`

## Проблема

План реструктуризации предполагал:
1. Перенос `src/` → `apps/web/src/` (фронт в отдельной папке).
2. Переименование `backend-laravel/` → `core/` (ядро API в отдельной папке).

Обе операции физически ломают ключевые сценарии проекта:

### Фронт (`src/` → `apps/web/`)
- **Lovable-платформа** жёстко ожидает Vite-проект в корне: `package.json`, `index.html`, `vite.config.ts`, папку `src/`. Пайплайн авто-деплоя (Lovable → GitHub → прод) читает корневой `package.json` и запускает `bun run build` с корня.
- Внутренние инструменты Lovable (auto-context, runtime errors, shadcn CLI через `components.json`, security scanner) индексируют пути `src/...`. Перенос ломает трассировку ошибок и подсказки.
- Требование пользователя: **разработку можно вести и через Lovable, и локально через обычный workflow**. Локальный dev при этом работает (`bun dev` с корня), а Lovable-деплой сохраняется.

### Ядро (`backend-laravel/` → `core/`)
- Продовый деплой: `deploy/deploy-laravel.sh` жёстко ссылается на `backend-laravel/`.
- `docker-compose.yml` и `docker-compose.sandstorm.yml` монтируют `./backend-laravel:/var/www/html`.
- CI-workflow (`.github/workflows/*`) собирает артефакт из `backend-laravel/`.
- Все существующие ссылки в документации, тестах и скриптах.

Переименование = одномоментный флаг-дэй с риском положить прод.

## Решение

**Физическая структура остаётся текущей:**
```
/                    ← фронт (Vite + React), корневой package.json
/backend-laravel/    ← ядро API (Laravel), исторически "core"
/services/*/         ← микросервисы (пока каркас документации)
```

**Логическая (документационная) структура — целевая:**
```
/apps/web/           ← rtfm_web.md, .env.example (указатели на корень)
/core/               ← rtfm_core_api.md, .env.example (указатели на backend-laravel/)
/services/*/         ← rtfm_<name>.md, .env.example, будущий код
```

Папки `apps/web/` и `core/` содержат **только** документацию (`rtfm_*.md`) и `.env.example`. Они описывают *целевую* архитектуру для внешних разработчиков и служат местом, куда постепенно перенесётся ownership: `core/rtfm_core_api.md` — единственный источник правды по API-контракту, даже пока код в `backend-laravel/`.

## Последствия

**Плюсы:**
- Lovable-пайплайн работает без изменений — правки через чат по-прежнему коммитятся и деплоятся.
- Локальные разработчики могут работать через любой IDE (`bun install && bun dev` из корня, `composer install` в `backend-laravel/`).
- Прод-деплой не ломается.
- Документация уже структурирована по будущей раскладке — переезд, когда он созреет, будет чисто механическим.

**Минусы:**
- Двойственность: физический путь ≠ логическое имя. Смягчается тем, что каждый `rtfm_*.md` явно указывает физическое расположение.
- Внешние разработчики должны прочитать этот ADR перед клонированием репозитория.

## Условия физического переезда (будущее)

Переезд `src/` → `apps/web/` и `backend-laravel/` → `core/` возможен только при одновременном выполнении:
1. Lovable-платформа поддерживает монорепо-режим (`workspaceRoot: "apps/web"`).
2. Или: команда осознанно отказывается от Lovable-редактирования и переходит на чистый git-workflow.
3. Обновлены: `deploy/*.sh`, `docker-compose*.yml`, `.github/workflows/*`, все `README.md`.

До этого момента — не трогаем.
